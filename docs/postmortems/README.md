# postmortems — 장애 회고와 재발 방지 (개발 후)

**개발 "후" 단계.** 운영 중 실제로 터진 장애를 기록하고, *같은 일이 다시 나지 않게 스펙·테스트·훅으로 못박는 곳*이다.

원칙은 blameless: "누가 잘못했나"가 아니라 **"어떤 가드레일이 없어서 샜고, 무엇을 바꿔 막았나"** 를 남긴다.

## 무엇을 여기에 (그리고 무엇을 안 두나)

- **여기:** 운영/alpha에서 실제 발생한 장애·사고 (사용자 영향 있음).
- **여기 아님:** 계획된 AI 한계 실험 → [experiments/ai-limits-experiments.md](../experiments/ai-limits-experiments.md). 주간·48시간·2개월 정기 회고 → [operations/](../operations/).
- **경계 사례:** AI가 만든 결함이 운영까지 새어 장애가 됐다면 → 여기에 postmortem으로 남기고 실험 서사와 교차 링크.

파일명: `YYYY-MM-DD-제목.md` (예: `2026-08-14-digest-double-send.md`)

## 템플릿 (복사해서 채운다)

```markdown
---
제목: <무슨 장애>
발생: YYYY-MM-DD HH:MM ~ HH:MM (KST) / 심각도: SEV1 | SEV2 | SEV3
영향: <누가·무엇을·얼마나>
상태: 조사중 | 종결
---

## 한 줄 요약
<무엇이, 왜, 얼마나.>

## 타임라인 (탐지 → 완화 → 복구)
- HH:MM <일어난 일>

## 근본 원인 (왜 일어났나 — 표면 원인에서 5 whys로 내려간다)

## 어디서 탐지됐나
<test / CI / alert / 사용자 문의 중 어디서 처음 잡혔나. 못 잡았다면 왜 못 잡았나가 핵심.>

## 영향 범위 (지표·건수)
<영향받은 사용자·요청·데이터. 감이 아니라 숫자로.>

## 즉시 조치 (완화)

## 재발 방지 (반드시 실행 가능한 항목)
- [ ] 스펙/테스트/훅/알림 중 무엇을 바꿨나 → (링크)
- [ ] 같은 유형을 재현하는 회귀 테스트 추가 → (링크)
- [ ] 탐지가 늦었다면 alert/지표 보강 → (링크)

## 배운 것 / 남길 증거
<이 사건을 면접에서 "지표·장애를 보고 결정을 바꾼 경험"으로 어떻게 말할지 한 줄.>
```

> 좋은 postmortem은 "고쳤다"에서 끝나지 않고 **가드레일의 결함과 개선**까지 드러낸다.
> cs-learning G섹션(운영 회고)·A섹션(AI 실패→복구 서사)의 증거가 여기서 완성된다.
