## 1. 서비스 개요

### 한 줄 정의

> 링크를 저장하는 순간과 주기적으로, AI가 능동적으로 연관 콘텐츠를 꺼내주는 링크 아카이브 서비스

### 타겟 사용자

개발자/지식노동자. "나중에 읽으려고 저장했다가 결국 안 읽음" 을 직접 겪는 사람.  
디스콰이엇 런칭 타겟.

## 2. 문제 정의

### 핵심 문제

> 링크를 저장하는 행위가 "나중에 볼게"라는 의도인데,  
> 나중에 찾지도 못하고, AI가 먼저 꺼내주지도 않아서  
> 결국 링크는 쌓이기만 하고 지식으로 전환되지 않는다.

### 세부 문제

| # | 문제 | 발생 시점 |
|---|---|---|
| A | 저장하고 안 읽음 (무덤 문제) | 저장 후 시간 경과 |
| B | 유사 링크 중복 저장, 연결 기회 놓침 | 저장 순간 |
| C | 키워드 기억 안 나면 못 찾음 (검색 문제) | 검색 시 |
| D | 유형 구분 없이 한 통에 섞임 | 전반적 |

### 해결 범위

- **A + B:** 저장 시 연관 링크 즉시 제시 + 주간 다이제스트
- **C:** 자연어 시맨틱 검색
- **D:** AI 자동 카테고리 분류

## 3. 링크 유형 분류

| 유형 | 정의 | 예시 | AI 처리 |
|---|---|---|---|
| **콘텐츠형** | 특정 게시글/포스팅의 내용이 좋아서 저장 | 블로그, 미디엄, 벨로그, 커뮤니티 글 | 크롤링 + 요약 + 임베딩 + 분류 |
| **도구형** | 사이트 자체가 유용해서 접근용으로 저장 | 유용한 서비스 메인 URL | 분류만 |
| **레퍼런스형** | 반복 참고용 문서/스펙 저장 | 공식 문서, GitHub 레포, 논문 | 목차 요약 + 분류 |

> **핵심:** 유형별로 처리 파이프라인이 달라야 함. 도구형에 AI 요약은 낭비.

## 4. 핵심 기능 명세

### 4-1. 링크 저장 (크롬 익스텐션)

- 현재 탭 URL 원클릭 저장
- 저장 즉시 → 유형 자동 분류 (콘텐츠/도구/레퍼런스)
- 저장 즉시 → 연관 링크 팝업 노출 ("2달 전에 이것도 저장했어요")
  - 기준: 벡터 유사도 80% 이상 && 7일 이상 미열람 링크
  - 최대 3개 노출

### 4-2. 자연어 시맨틱 검색 (웹 앱)

- "그때 본 Redis 분산락 글" → 의미 기반 검색
- Qdrant 벡터 유사도 검색
- 키워드 기억 없어도 검색 가능
- MVP 디폴트 기능

### 4-3. 주간 다이제스트 (이메일/푸시)

**발송 기준:**
1. 콘텐츠형 링크 중 14일 이상 미열람 전체 추출
2. 주제별 클러스터링
3. 클러스터당 AI가 대표 1~2개 선별 + 묶음 3줄 요약
4. 최대 5개 클러스터 발송
5. 발송 주기: 매주 월요일 오전

**다이제스트 UI 구조:**
```
이번 주 안 읽은 것들 📬

[Kafka] 3개 저장하셨어요
→ "Kafka KRaft 모드 완전 정리" 외 2개
→ 핵심: ...
[열어보기] [다음 주에] [더보기 →]

[Spring Security] 2개 저장하셨어요
→ ...
```

**버튼 동작:**
- `열어보기` → 링크 URL 이동 + 자동 읽음 처리
- `다음 주에` → 7일 스누즈, 다음 다이제스트에 재포함
- `더보기 →` → 웹 아카이브 페이지로 이동 (해당 카테고리 전체 + 전체 탐색)

**읽음 처리 기준 (Option B):**
- 링크 URL 클릭 = 자동 읽음 처리
- 별도 "읽음" 버튼 없음
- 읽음 처리된 링크는 다이제스트에서 제외

### 4-4. 웹 아카이브 페이지

- 전체 링크 목록 (카테고리별 탐색)
- 시맨틱 검색
- 다이제스트 더보기 랜딩 포인트
- 이메일 매직링크로 자동 로그인 진입

## 5. 기술 선택 ADR

### 5-1. 메시지 큐: Kafka KRaft

**현재 상황:**  
링크 저장 이벤트 하나에 크롤링 → AI 요약 → 임베딩 → 분류 → 태깅 총 5단계 처리가 필요하고, 각 단계는 처리 시간과 실패 패턴이 다름.

**문제:**
- 동기 직렬 처리 시 응답 10~30초 (LLM 호출 포함)
- 하나 실패 시 전체 실패
- AI 모델 교체 시 재처리 로직 별도 필요

**선택지:**

| 옵션 | 장점 | 단점 |
|---|---|---|
| 동기 REST 직렬 | 구현 단순 | 응답 30초, 전파 실패 |
| Spring @Async | 즉시 응답 | 재시작 시 유실, 재처리 불가 |
| **Kafka KRaft** | 메시지 보존, 독립 재시도, 재처리 | 운영 복잡도 증가 |

**결정: Kafka KRaft**

결정적 이유:
1. 링크 저장 이벤트 하나를 크롤링·요약·임베딩·분류 컨슈머가 **각각 독립적으로 읽어야 함** → RabbitMQ는 한 컨슈머가 메시지 가져가면 사라지므로 구조 자체 불가
2. AI 모델 교체 시 임베딩 컨슈머만 **오프셋 되돌려 전체 재처리** 가능 → RabbitMQ는 소비된 메시지 없어 불가
3. 서버 재시작 시 처리 중 작업 **유실 없음** → @Async는 메모리에만 존재

**KRaft 선택 이유 (Zookeeper 제외):**
- Kafka 4.0부터 Zookeeper 공식 제거, 현재 표준
- 메모리 ~1GB 절약 (Zookeeper ~1GB 불필요)
- 단일 프로세스로 운영 단순화

**트레이드오프:**

| 트레이드오프 | 영향도 | 해결 방안 |
|---|---|---|
| 처리 완료 시점 클라이언트 전달 복잡도 | 중간 | 저장 즉시 `status: PROCESSING` 응답, 상세 진입 시 polling 또는 WebSocket으로 `READY` 수신 (VODA 2-Phase 패턴 재활용) |
| 단일 인스턴스에서 Kafka 운영 복잡도 | 낮음 | Kafka 핵심 가치는 서버 수가 아닌 단계 간 디커플링 + 메시지 내구성. 단일 인스턴스에서도 유효 |

**치명적 트레이드오프 없음.**

---

### 5-2. Kafka vs RabbitMQ

| 항목 | RabbitMQ | Kafka |
|---|---|---|
| 메시지 소비 후 | 삭제 | **보존** |
| 재처리 | 불가 | **오프셋 되돌리기** |
| 다중 컨슈머 독립 소비 | 불가 | **각 컨슈머 그룹이 독립 읽기** |
| 적합 용도 | 단순 작업 큐 | **이벤트 스트림/파이프라인** |

**이 서비스에서 RabbitMQ 불가 이유:**  
크롤링·요약·임베딩 컨슈머가 동일 이벤트를 각각 독립 소비해야 하는 구조 자체가 RabbitMQ로 불가능.

### 5-3. 검색: Vector DB (Qdrant) + LLM Wiki (Phase 2)

**역할 분리:**

| | Vector DB (Qdrant) | LLM Wiki |
|---|---|---|
| AI 호출 시점 | 저장 시 (임베딩 생성) | 배치 주기 (위키 업데이트) |
| 조회 시 AI | 불필요 (유사도만) | 불필요 (마크다운 직독) |
| 저장 결과물 | 숫자 벡터 | 정리된 마크다운 |
| 강점 | "이것과 비슷한 링크 찾기" | "이 주제 지식 정리 + 갭 발견" |

**Phase 1 (MVP):** Qdrant만 사용
- 저장 시 연관 링크 제시
- 시맨틱 검색

**Phase 2 (런칭 후):**  
저장된 링크들을 LLM이 배치로 읽어 위키 페이지 생성 → 주제별 갭 발견 → 알림

### 5-4. 인증: 소셜 로그인 + JWT

**크롬 익스텐션 인증 플로우:**
```
첫 설치
→ 팝업에서 소셜 로그인 (Google/GitHub OAuth)
→ JWT + RefreshToken 발급
→ chrome.storage.local 저장 (브라우저 재시작 후에도 유지)
→ 링크 저장 시 Authorization: Bearer {JWT} 포함 API 호출
→ 401 수신 시 RefreshToken으로 자동 재발급
```

**이메일 다이제스트 → 웹 앱 진입:**
```
다이제스트 "더보기" 링크
→ 단기 매직링크 토큰 파라미터 포함
→ 웹 앱 자동 로그인 진입 (별도 로그인 불필요)
```

## 8. Kafka 파이프라인 상세

```
POST /api/links (링크 저장 요청)
→ DB에 status: PROCESSING으로 즉시 저장
→ 200 응답 반환 (사용자 대기 없음)
→ LinkSavedEvent 발행

컨슈머 처리 (독립 병렬):
1. 유형분류 컨슈머
   → 콘텐츠형 / 도구형 / 레퍼런스형 판별
   → DB status 업데이트

2. 크롤링 컨슈머 (콘텐츠형만)
   → 본문 수집 + 스냅샷 저장 (dead link 방지)

3. AI요약 컨슈머 (콘텐츠형만)
   → 3줄 요약 생성

4. 임베딩 컨슈머 (콘텐츠형 + 레퍼런스형)
   → 벡터 생성 → Qdrant 저장
   → status: READY

클라이언트:
→ 저장 직후 PROCESSING 상태 표시
→ polling 또는 WebSocket으로 READY 수신 시 연관 링크 팝업
```
