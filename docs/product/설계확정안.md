# LinkPocket 설계 확정안

> 작성일: 2026-07-01 · 최종 갱신: 2026-07-15
> 상태: **확정 — MVP 제품 범위의 단일 기준**
> 기술 선택과 런타임 설계는 [decisions/](../decisions/)·[architecture/](../architecture/)가, 기능별 구현 계약은 [plan/](../plan/)이 기준이다.

## 1. 서비스 개요

> 저장한 링크를 나중에 찾고 다시 열 수 있게, AI가 요약·검색·연관 추천을 제공하는 링크 아카이브 서비스

대상은 개발자와 지식노동자다. "나중에 읽겠다"며 저장한 링크가 묻히고, 키워드가 기억나지 않아 다시 찾지 못하며, 이미 저장한 유사 자료를 또 저장하는 문제를 해결한다. 디스콰이엇 공개를 초기 런칭 대상으로 둔다.

## 2. 해결할 문제와 방법

| 문제 | LinkPocket의 해결 방식 |
|---|---|
| 저장 후 다시 열지 않음 | `openedAt` 기반 재발견과 P1 주간 다이제스트 |
| 유사 링크를 중복 저장하고 연결을 놓침 | P0 저장 완료 뒤 연관 링크 최대 3개 추천 |
| 기억나는 키워드가 없어 검색하지 못함 | P0 자연어 시맨틱 검색 |
| 개인 분류 체계가 없음 | P0 사용자가 만드는 카테고리와 다중 분류 |
| 페이지 수집·요약이 실패하면 저장 자체도 실패 | URL·제목을 우선 보존하고 가능한 메타데이터만 추가 |

## 3. MVP의 링크·카테고리 모델

MVP에서 모든 저장 대상은 하나의 **Link**다. 콘텐츠형·레퍼런스형·도구형 같은 별도 링크 유형을 만들거나 사용자에게 선택받지 않는다. 사용자는 `레퍼런스`, `도구`, `Backend`처럼 원하는 카테고리를 직접 만들고 하나의 링크에 여러 카테고리를 붙인다.

- 카테고리 생성·수정·삭제와 링크의 다중 카테고리 연결은 P0다.
- 카테고리 삭제는 링크를 삭제하지 않고 연결만 해제한다.
- 저장 시 카테고리는 선택 사항이며 미분류 상태를 허용한다.
- 유형별 처리 차이가 실제 사용·운영 지표로 필요해질 때만 내부 `linkKind`와 별도 파이프라인을 재검토한다.

## 4. P0 기능

### 4-1. 링크 저장과 최소 보존

Chrome 확장 프로그램에서 현재 탭을 저장한다. 저장 요청은 빠르게 성공해야 하며, 수집·요약·색인은 백그라운드에서 처리한다.

저장 성공의 최소 보장 값은 다음과 같다.

- 원본 URL과 정규화한 canonical URL
- 제목: 탭 제목 → Open Graph 제목 → hostname/path 순으로 fallback해 비어 있지 않게 보장
- 사용자, 생성 시각, 처리 상태

가능하면 OG 설명·대표 이미지·추출 본문·AI 요약·임베딩을 추가한다. paywall, SPA, 로그인 필요 페이지 등으로 수집이 실패해도 링크는 보존한다. 처리 결과는 `QUEUED`, `PROCESSING`, `READY`, `READY_WITHOUT_CONTENT`, `FAILED`처럼 사용자 행동에 필요한 상태만 노출한다.

### 4-2. 수집·AI 요약·사용자 보정

모든 Link에 대해 안전한 URL 검증 뒤 본문 추출, 3줄 내외 AI 요약, 임베딩 생성을 시도한다. 추출 품질은 고정 URL 세트에서 parser와 fallback을 비교 실험해 개선한다.

- 사용자는 제목·AI 요약·카테고리를 수정할 수 있다.
- 사용자 수정값은 이후 재수집·재요약이 자동으로 덮어쓰지 않는다.
- 검색용 임베딩에 제목·요약을 포함했다면 수정 뒤 해당 Link만 재색인한다.

### 4-3. 저장 직후 연관 링크 추천

색인이 준비되면 클라이언트 polling으로 완료 상태를 확인하고, 새 Link와 의미적으로 가까운 기존 Link를 최대 3개 보여 준다. 초기 기준은 유사도 80% 이상이면서 7일 이상 열지 않은 Link다. 이 추천은 P0이며 제품의 핵심 차별점이다.

### 4-4. 웹 아카이브와 자연어 검색

- 개인별 전체 Link 목록과 카테고리별 탐색
- 자연어 시맨틱 검색: 예) "그때 본 Redis 분산락 글"
- 검색 결과에서 제목, AI 요약, 원문 링크, 카테고리, 처리 상태 제공
- 다이제스트의 더보기 랜딩과 이메일 매직링크 진입

검색은 저장된 링크를 찾아 보여 주는 retrieval 기능이다. 링크 내용을 근거로 새 답변을 생성하는 RAG 질의응답 기능은 MVP 범위에서 제외한다.

### 4-5. 열람 기록

LinkPocket이 유도한 외부 링크 열기만 기록한다. 실제로 읽었음을 단정하지 않으므로 `readAt` 대신 `openedAt`을 사용한다.

- 아카이브·검색·연관 추천·다이제스트의 외부 링크는 LinkPocket redirect를 거쳐 원문으로 이동하며 `openedAt`, `openCount`, `source`를 기록한다.
- `source`는 `archive`, `search`, `related`, `digest`로 구분한다.
- Chrome 확장 프로그램 밖에서 사용자가 직접 방문한 기록은 추적하지 않는다.

## 5. P1 — 주간 다이제스트 이메일

다이제스트는 **주간 미열람 링크를 주제별로 다시 꺼내 주는 이메일**이다. 모든 Link가 후보이며 링크 유형으로 제외하지 않는다.

매주 월요일 오전에 다음을 수행한다.

1. 14일 이상 `openedAt`이 없는 Link 중 snooze·영구 제외가 아닌 후보를 찾는다.
2. 임베딩으로 주제별 클러스터를 만들고, 클러스터마다 대표 Link 1~2개와 묶음 요약을 만든다.
3. 최대 5개 클러스터를 이메일로 보낸다.

```text
이번 주 안 열어 본 링크 📬

[Kafka] 3개 저장했어요
→ "Kafka KRaft 모드 완전 정리" 외 2개
→ 핵심: ...
[열어보기] [다음 주에] [더보기]
```

- `열어보기`: 원문으로 이동하고 `openedAt` 기록
- `다음 주에`: 7일 snooze 후 다음 다이제스트 후보로 유지
- `안 볼래요`: 다이제스트에서 영구 제외
- `더보기`: 매직링크로 웹 아카이브의 해당 카테고리/탐색 화면 진입

이메일 보안 스캐너가 링크를 선행 요청할 수 있으므로 다이제스트 click 수치는 추정치로 취급하고, 열람 기록의 출처와 실제 재방문 지표를 분리해 해석한다.

## 6. 후순위·조건부 범위

- 링크 유형(`linkKind`) 분리와 유형별 특화 파이프라인
- 다이제스트 클러스터링 품질 고도화, 사용자별 발송 설정·개인화
- 푸시 알림, 모바일 저장 진입점
- LLM Wiki·지식 공백 진단
- 답변 생성형 RAG
- Kafka, Redis, SSE, Qdrant, headless browser: 사용량·품질·운영 신호가 확인될 때만 [조건부 기술 도입 판정](../decisions/conditional-tech-adoption.md)에 따라 검토

## 7. MVP 우선순위

| 우선순위 | 기능 |
|---|---|
| P0 | Chrome 확장 프로그램 링크 저장, Google 로그인, 최소 보존·처리 상태 |
| P0 | 사용자 카테고리 CRUD와 다중 분류 |
| P0 | 안전한 수집, AI 요약, 임베딩, 사용자 제목·요약 보정 |
| P0 | 웹 아카이브, 자연어 시맨틱 검색 |
| P0 | 저장 완료 뒤 연관 링크 추천, `openedAt` 기록 |
| P1 | 주간 다이제스트 이메일 |
| 후순위 | 유형 분리, 답변 생성형 RAG, Wiki, 푸시·개인화 |

## 8. 구현 전 산출물

- ERD: User, Link, Category, LinkCategory, Content, Chunk, Job, Digest, OpenEvent
- OpenAPI: 저장·처리 상태·카테고리·검색·연관 추천·열람 redirect·다이제스트·인증
- 상태 머신: 수집/색인과 삭제 파이프라인, 재시도·멱등 규칙
- 파서·요약 실험: 세 유형 성격의 URL 표본에서 추출 성공률·요약 충실도·비용 비교
- 기능별 plan: acceptance criteria, 불변식, 실패 조건을 먼저 확정
