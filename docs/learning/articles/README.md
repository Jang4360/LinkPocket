# articles — 학습 아티클 (작성 가이드 + 템플릿)

정해진 범위의 주제 하나를, 면접에서 그대로 말할 수 있는 완성본으로 정리하는 곳.
파일명: `주제.md` (kebab-case, 예: `spec-contract-tdd.md`, `ssrf-url-fetch.md`, `at-least-once-idempotency.md`)

## 쓰기 전 (순서 고정)

1. [../cs-learning.md](../cs-learning.md)에서 다룰 **축(A~H)** 과 그 축의 `검증·기록할 증거` 칸을 확인한다.
2. [../README.md](../README.md)의 원칙을 확인한다.
3. 아래 **말투·품질 규칙**을 지키며 **템플릿**을 채운다.

## 주제 고르는 법

- **질문형 금지.** 제목은 물음표로 끝내지 않는다. 범위가 정해진 명사구여야 한다.
  - ✅ `명세·계약과 TDD 기반 테스트 자동화`, `dense vs BM25 vs hybrid retrieval`
  - ❌ `왜 테스트는 어려운가`, `캐시는 왜 힘든 문제인가`
- 한 아티클 = cs-learning의 한 축, 또는 그 축 안의 뚜렷한 소항목 하나. 두 축을 한 글에 섞지 않는다.

## 말투·품질 규칙 (레퍼런스에서 뽑은 것)

- **정의가 아니라 장면으로 연다.** 추상 정의로 시작하지 말고, LinkPocket에서 이 주제가 실제로 문제/필요가 되는 구체적 순간부터. "정상처럼 보이지만 사실은…" 식으로 증상과 오해를 먼저 보여준다.
- **한 줄 명제로 관점을 튼다.** 여는 장면 뒤에 blockquote 한 줄: *"X는 ~하는 기술이 아니라, ~하는 일이다."* 흔한 오해를 재정의로 교정한다.
- **"A가 아니라 B다" 프레이밍**을 즐겨 쓴다. (예: "조회 속도 문제가 아니라 복사본 생명주기 문제")
- **핵심 용어는 `backtick`.** (`Red-Green-Refactor`, `Testcontainers`, `escaped_defect`)
- **개념은 아티팩트로 고정한다.** 코드·설정·테스트·JSON 스니펫 하나로 추상을 붙잡는다.
- **반드시 운영 신호·지표를 준다.** 감이 아니라 숫자로 판단하게. 지표 이름도 `backtick`.
- **트레이드오프는 표로.** `선택지 | 좋아지는 것 | 비싸지는 것`.
- **긴장 관계를 굵게 못박고 닫는다.** "둘을 동시에 최대화할 수 없다"류의 문장으로. 정답을 강요하지 말되 "이 상황엔 이걸"이라는 현실적 권고는 준다.
- **과장·마케팅 금지, 정직하게.** 완화책은 "완화책"이라 부르고, 차이가 근소하면 근소하다고 쓴다.
- 분량은 스크롤 3~5번. 각 섹션 3~6문장. 길이보다 밀도.

cs-learning 축: <A~H 중 하나 + 소항목 — 예: B. 명세·계약·테스트 / Red-Green-Refactor>

<여는 장면: LinkPocket에서 이 주제가 실제로 문제/필요가 되는 순간 2~3문장.
"정상처럼 보이지만 사실은…" 으로 증상과 흔한 오해를 먼저.>

> <한 줄 명제: "X는 ~하는 기술이 아니라, ~하는 일이다." — 재정의로 관점 전환.>

<주제의 범위를 못박는다. 무엇을 포함하고 무엇은 포함하지 않는지. 핵심 용어는 `backtick`.>

## 어떻게 동작하는가

<특히 AI가 생성한 코드에서 자주 나는 실패를 최소 하나 포함한다 (이 프로젝트의 축).>

## 채운 예시 (헤더만 — 감을 잡기용)

주제: **명세·계약과 TDD 기반 테스트 자동화** (cs-learning B축)

- **여는 장면** — AI에게 "링크 저장 API 만들어줘"라 했더니 구현과 테스트를 한 번에 뱉었고, 테스트는 초록불인데 중복 저장 시 row가 2개 생긴다. 통과처럼 보이지만 계약이 없었다.
- **한 줄 명제** — *"TDD는 테스트를 먼저 짜는 절차가 아니라, AI가 무엇을 만족시켜야 하는지 사람이 먼저 못박는 일이다."*
- **개념 자리잡기** — `Red-Green-Refactor`, 테스트 더블, 계약(OpenAPI)·불변식의 범위. 아티팩트: URL 정규화의 실패 테스트 스니펫.
- **어떻게 동작하는가** — 실패 테스트로 행동을 고정 → 최소 구현 → 리팩터 / 사람이 계약·대표 실패 테스트를 정하고 AI는 구현·추가 케이스만.
- **어디서 무너지는가** — AI에게 구현과 테스트를 동시에 맡기고 초록불만 확인 → 멱등성 같은 불변식이 spec에 없어 회귀가 샌다.
- **무엇을 보고 판단하는가** — `escaped_defect`, red→green→refactor 커밋 비율, CI 차단 수, mutation score. "테스트 통과율"만 보면 숨는다.
- **선택지가 있다면** — 단위/Testcontainers 통합/계약 테스트의 속도 vs 신뢰도 트레이드오프 표. **"빠른 피드백과 실제 SQL 검증을 한 계층으로 동시에 얻을 수 없다."**
- **적용·증거** — 1주차 인증+저장 vertical slice / 실패 테스트 커밋 3~5개 + 계약 CI 로그 / "AI 생성 코드를 계약과 실패 테스트로 통제했다."

발행용 초안도 여기서 쓰고, 발행 링크는 [../../reference/](../../reference/)에 기록한다.
