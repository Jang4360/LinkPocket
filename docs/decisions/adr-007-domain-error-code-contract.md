# ADR-007: 도메인별 비즈니스 에러 코드 계약 (BE-FE 합의)

- 날짜: 2026-07-16 / 상태: 확정
- **범주: 아키텍처**
- 관련: [architecture/api-error-contract.md](../architecture/api-error-contract.md), [plan-01-auth-google-oauth.md](../plan/01-auth-google-oauth.md)

## 상황

실제 API 구현(plan-01의 첫 엔드포인트)에 들어가기 전, "예외가 났을 때 프론트에 무엇을 내려주고 화면이 무엇을 보여줄지"가 정해지지 않았다. 이걸 나중에 API마다 즉흥적으로 정하면 ① 같은 실패가 도메인마다 다른 형태로 내려가고 ② 프론트가 `message` 문자열을 파싱해 분기하게 되며 ③ 화면 처리(토스트/리다이렉트/인라인 에러) 합의가 코드에 흩어진다.

## 선택지

| 옵션 | 장점 | 단점 |
|---|---|---|
| HTTP status만으로 분기(401→로그인, 404→없음 등) | 구현 단순 | 같은 401도 "세션 만료"와 "다른 계정 세션 충돌"은 화면 처리가 다른데 구분 불가 |
| 전역 단일 에러코드 enum(`ErrorCode` 하나) | 코드 하나로 관리 편함 | 도메인이 늘수록 거대해지고, 도메인 소유권이 불명확해짐(누가 어떤 코드를 추가해도 됨) |
| **도메인별 enum + 공통 인터페이스 + 공통 응답 envelope** (채택) | 도메인이 자기 에러를 소유, 컴파일 타임에 강제, envelope는 FE가 일관되게 파싱 | 도메인마다 enum 파일이 늘어남(관리 비용은 오히려 장점 — 응집도) |

## 결정

**도메인별 에러 코드 enum + 공통 `ErrorCode` 인터페이스 + 공통 JSON envelope.** 상세 스펙은 [architecture/api-error-contract.md](../architecture/api-error-contract.md)에 두고, 모든 plan은 자기 도메인의 에러 코드 표를 이 스펙을 따라 자기 plan 문서에 채운다.

핵심 설계:
- 에러 코드는 `{DOMAIN}_{REASON}` 네이밍(예: `AUTH_SESSION_EXPIRED`, `AUTH_REFRESH_TOKEN_REUSED`).
- 각 도메인은 `ErrorCode`를 구현하는 자기 enum을 가진다(예: `AuthErrorCode`). 새 실패 유형을 추가하려면 그 도메인 enum에 상수를 추가해야 하므로, **컴파일 타임에 강제**되고 다른 도메인과 섞이지 않는다.
- HTTP status와 비즈니스 코드는 **분리**한다 — HTTP status는 범용 처리(재시도 가능 여부, 인증 필요 등)를, 비즈니스 코드는 화면별 구체적 처리를 담당한다.
- **FE-BE 계약 표**(코드 → HTTP status → 화면 처리 → 사용자 문구)를 각 plan 문서에 채워, 코드 리뷰 시점에 프론트 처리까지 같이 확정한다.

## 트레이드오프

- 도메인마다 enum 파일이 늘어난다 → 도메인 응집도 관점에선 오히려 바람직(그 도메인 담당자/plan이 자기 에러를 소유).
- 초기 설계 비용이 있다(공통 인터페이스·전역 예외 핸들러를 먼저 만들어야 함) → plan-01의 task로 편입해 한 번만 만들고 이후 도메인은 enum만 추가.

## 재검토 조건

- 도메인이 10개 이상으로 늘어 코드 검색·문서화가 부담되면 코드 레지스트리(전체 코드 자동 수집 문서화 스크립트)를 도입한다.
- FE가 실제로 `message` 문자열에 의존하는 패턴이 반복 발견되면 envelope에 필드를 보강한다.

## 근거

- 사용자 요청: "도메인별로 같은 에러라도 비즈니스 코드 enum 타입이 달라야 한다 + FE와 합의하는 계약 문서가 있어야 한다."

## 면접 답변 요지

> "에러 코드를 전역 enum 하나로 두지 않고 도메인별 enum + 공통 인터페이스로 분리해, 각 도메인이 자기 실패 유형을 소유하고 컴파일 타임에 강제되게 했다. HTTP status와 비즈니스 코드를 분리해 범용 처리와 화면별 처리를 나눴고, 코드마다 화면 처리 방식을 표로 만들어 프론트와 API 구현 이전에 합의했다."
