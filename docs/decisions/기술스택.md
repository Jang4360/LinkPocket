## 0. 이 문서를 쓴 이유

LinkPocket의 8주 개발이 이번 주(0주차)부터 시작된다. 다음 세션에서 개발을 이어받을 때, 그리고 채용 면접에서 "왜 이 스택인가"라는 질문을 받을 때, 감이 아닌 **근거**로 대답할 수 있어야 한다.

기술 선택의 원칙은 세 가지다.

1. **채용 반복 신호에 맞는가**: 2025+ 국내 백엔드 신입 채용에서 실제로 요구/우대하는가.
2. **CS 학습 축(P0)과 자연스럽게 연결되는가**: 매트릭스의 P0가 이 스택에서 실습 가능한가.
3. **개인 프로젝트 규모에서 운영 가능한가**: $0 인프라 예산과 하루 6시간 개발 시간에 무리가 없는가.

이 세 가지를 통과하지 못한 후보는 P1 실험으로 미루거나 기각했다.

## 2. 각 선택의 근거

### 2-1. Java 21 LTS + Gradle Kotlin DSL

**대안과 이걸 택한 핵심 이유**
- Java 17 LTS + Groovy: Virtual Threads가 21에서 정식(JEP 444) → 17로는 이 프로젝트의 핵심 실습(크롤링 I/O 동시성) 자체가 불가. (핵심: Virtual Threads 정식)
- Java 21 + Maven: 빌드 스크립트를 Kotlin DSL로 두면 AI가 만든 잘못된 설정이 컴파일에서 걸림 → Maven XML엔 그 타입 안전이 없음. (핵심: 빌드 설정 타입 안전)

**근거**
- Virtual Threads (JEP 444, Java 21 정식)는 크롤링·요약처럼 **I/O bound + 동시 요청 다수**인 워크로드에 정확히 맞는다. CS 매트릭스 D섹션(외부 URL 수집)·E섹션(비동기 파이프라인)의 실습 대상이다.
- 2025+ 국내 채용에서 Java 17 → 21 전환이 늘고 있다. 21로 시작하면 다음 3~5년간 재학습 부담이 없다.
- Gradle Kotlin DSL은 IDE 자동완성·타입 검증이 강해 AI 코딩 하네스와 궁합이 좋다 (에이전트가 잘못된 설정 쓰면 컴파일 시점에 잡힘).

**트레이드오프**
- 일부 서드파티 라이브러리가 Java 21 검증 안 됐을 수 있음 → 문제 발생 시 개별 이슈로 대응.
- Groovy DSL 예제가 여전히 더 많음 → Kotlin DSL 마이그레이션 문서 참고 필요.

**재검토 조건**
- 특정 라이브러리가 Java 21에서 미해결 이슈로 개발을 막을 때만 21 유지 + 해당 부분 별도 해결.

**면접 답변 요지**
> "I/O 다중 요청이 많은 크롤링 파이프라인에서 Virtual Threads로 platform thread 튜닝 없이 자연스러운 동시성을 실습하려 21을 선택했다. Gradle Kotlin DSL은 AI 하네스가 잘못된 빌드 설정을 만들었을 때 컴파일 시점에 잡히는 장점이 있다."

### 2-2. Spring Boot 3.4.x

**대안과 이걸 택한 핵심 이유**
- Spring Boot 3.3.x: 매직링크에 쓸 Spring Security 6.4 One-Time Token과 Micrometer OTLP 성숙이 3.4 라인에 들어옴 → 3.3은 이 둘을 직접 구현·백포트해야 함. (핵심: One-Time Token + OTLP)

**근거**
- Spring AI 어댑터 안정, Micrometer OTLP 지원, Spring Security 6.4의 One-Time Token 지원(매직링크에 활용).
- 3.4는 MockRestServiceServer 개선, 다양한 Testcontainers 개선 포함.

**트레이드오프**
- 신선한 버전이라 일부 서드파티 이슈 가능.

**재검토 조건**
- 라이브러리 호환 이슈 시 3.3.x로 강등.

---

### 2-3. JPA(Hibernate) + QueryDSL

**대안과 이걸 택한 핵심 이유**
- JPA + jOOQ: 순수 SQL 표현력은 jOOQ가 낫지만, 국내 신입 채용이 압도적으로 JPA/QueryDSL 조합을 물어봄 → 채용 신호가 결정적. (핵심: 채용 신호)
- Spring Data JDBC 단독: 영속성 컨텍스트·연관관계·N+1 같은 JPA 학습 표면(채용 단골)을 못 다룸. (핵심: JPA 학습 표면)
- MyBatis: 국내 신호 감소 추세이고, 복잡 쿼리 타입 안전은 QueryDSL로 유사하게 확보. (핵심: 신호 감소 + QueryDSL 대체)

**근거**
- 국내 백엔드 신입 채용에서 **가장 자주 물어보는 조합**. 채용 시장 신호가 결정적이었다.
- 도메인 CRUD·연관관계는 JPA로 짧게, 복잡 쿼리(상태 머신 claim의 `SKIP LOCKED`, keyset pagination, EXPLAIN 실습)는 QueryDSL 타입 안전으로 표현.
- Hibernate 6.x는 pgvector 커스텀 타입 등록이 가능해 벡터 컬럼을 도메인 모델로 다룰 수 있음.

**트레이드오프**
- **N+1 함정이 상시 존재** → CS 매트릭스 D섹션의 "N+1과 fetch 전략" 학습이 실전 문제로 등장하는 점은 오히려 학습 관점에서 이득.
- 순수 SQL 직관성은 jOOQ가 강함.

**재검토 조건**
- 복잡 집계·배치 쿼리가 QueryDSL로 어색해지면 해당 영역만 jOOQ 부분 도입 (이건 5섹션 조건부 도입 표에 추가할 만함).

**면접 답변 요지**
> "채용 시장 신호와 도메인·복잡쿼리 분리 실습을 동시에 잡기 위해 선택했다. QueryDSL의 타입 안전은 AI 에이전트가 쿼리를 만들 때 잘못된 필드 참조를 컴파일에서 걸러낸다. N+1은 CS 학습 대상이므로 회피가 아니라 정면 실습으로 다룬다."

### 2-4. Neon PostgreSQL + pgvector (Qdrant 대신)

**대안과 이걸 택한 핵심 이유**
- Qdrant Cloud 별도: 링크·벡터 삭제를 하나의 트랜잭션으로 처리하고 관측·관리 대상을 1개 줄임. alpha 규모(100만 벡터 이하)에선 검색 품질 차이 없음. (핵심: 삭제 전파 단일 트랜잭션 + 인프라 표면 축소)
- self-hosted PostgreSQL: 6GB 인스턴스에 DB까지 얹으면 앱과 자원 경합. Neon 관리형이 백업·connection pooling 부담을 대신 가져감. (핵심: 인스턴스 자원 보호)

**근거**
- **이중 저장소를 하나로 통합**: 사용자 탈퇴·링크 삭제 시 DB row + 벡터가 하나의 트랜잭션에서 처리 가능. CS 매트릭스 D섹션의 삭제 파기 파이프라인이 훨씬 단순해진다.
- **인프라 표면 축소**: 관측 대상 1개 감소, 무료 tier 관리 대상 1개 감소.
- **성능 근거**: pgvector는 100만 벡터 이하에서 Qdrant와 유의미한 검색 품질/속도 차이가 없음 (커뮤니티 벤치마크 다수). LinkPocket alpha 규모는 이 범위 안.
- **실험 여지 확보**: 8주차 실험 후보로 "Qdrant 이관 시 검색 품질·비용 비교"를 남겨두면 포트폴리오 증거로 강함.

**트레이드오프**
- Qdrant의 hybrid retrieval(sparse+dense) API가 pgvector보다 직관적. pgvector는 BM25를 별도로(Postgres full-text search) 붙여 결합해야 함.
- pgvector는 HNSW/IVFFlat 인덱스 튜닝을 직접 학습해야 함 → 이건 F섹션 검색 품질과 자연스럽게 붙음(오히려 학습 이득).

**재검토 조건**
- 벡터 수 100만 초과 또는 hybrid retrieval SLO 위반 시 Qdrant 이관 실험.

**면접 답변 요지**
> "삭제 전파를 하나의 트랜잭션으로 처리하고 인프라 표면을 줄이려고 pgvector로 통합했다. 8주차 실험에서 Qdrant 이관 시 hybrid retrieval 품질·비용을 비교하는 걸 계획하고 있다. 벡터 수 100만 이하에서는 pgvector가 성능 면에서 밀리지 않는다는 벤치마크가 판단 근거였다."

### 2-5. Apache HttpClient 5

**대안과 이걸 택한 핵심 이유**
- Spring WebClient(Reactor): reactive 스택이 나머지(JPA blocking + Virtual Threads)와 패러다임 충돌. pool/timeout/SSRF를 설정으로 드러내는 표현력도 HttpClient 5가 더 명시적. (핵심: non-reactive 정합 + 명시적 설정)
- JDK HttpClient: per-route 연결 풀 상한·retry 분류·redirect SSRF 재검증 훅 같은 세밀 제어 API가 약함. (핵심: 세밀한 pool/timeout 제어)
- OkHttp: 인터셉터로 유사 구현이 가능해 결정적 차이는 크지 않음. connection pool·per-route·timeout 3종을 설정으로 명시하는 학습 목표에 HttpClient 5 API가 조금 더 곧이 맞아 택함. (핵심: 근소 — 학습 목표 정합)

**근거**
- CS 매트릭스 D섹션의 학습 목표(connection pool, per-host max, timeout 3종, keep-alive, retry 분류)를 **가장 명시적으로 설정하는 라이브러리**.
- SSRF 검증(redirect마다 host + resolved IP 재검증)이 자연스럽게 표현됨.
- Virtual Threads 위에서 platform thread 튜닝 없이 blocking I/O로 동시성 확보 가능.

**트레이드오프**
- 대용량 스트리밍 다운로드가 필요할 때 WebClient만큼 매끄럽지 않음 → 본문 크기 제한을 두므로 스트리밍 요구 자체가 크지 않음.
- 코드가 verbose함 → 유틸/설정 클래스로 감싸면 문제 없음.

**재검토 조건**
- 없음.

**면접 답변 요지**
> "connection pool과 timeout 3종을 설정값 암기가 아니라 도착률·downstream 한도로 계산하는 실습이 목표였다. HttpClient 5의 명시적 설정 API가 이 실습에 가장 잘 맞았다. Virtual Threads와 조합하면 blocking I/O로도 충분한 동시성이 나온다."

### 2-6. Spring Security OAuth2 Client (Google 단일)

**대안과 이걸 택한 핵심 이유**
- Google + GitHub 동시: 하나를 제대로 구현하면 재사용률이 높아 나중에 붙일 수 있음. 지금 둘을 동시에 하면 진입 마찰만 커짐. (핵심: MVP 마찰 최소 + 재사용)
- Keycloak/Auth0: 개인 프로젝트에 별도 IdP 운영·비용은 과함. Spring Security OAuth2 Client 표준 구현으로 충분. (핵심: 운영 오버헤드 과함)
- 매직링크 단일: 이메일 왕복 마찰이 있고 OAuth Code+PKCE 학습(H섹션 P0) 목표를 못 살림. 매직링크는 다이제스트 진입용으로 이미 씀. (핵심: OAuth 학습 표면 + 진입 마찰)

**근거**
- MVP 진입 마찰을 최소화. 대부분 개발자가 Google 계정을 보유하고 있어 Alpha 유입 저항이 작다.
- Spring Security OAuth2 Client는 표준 구현이 성숙하고 CS 매트릭스 H섹션(인증·인가) P0(OAuth 2.0 인가 코드+PKCE, state·redirect_uri 화이트리스트, JWT jti)를 그대로 실습 가능.
- Google 하나를 성공시키면 GitHub 추가는 코드 재사용률이 높아 alpha 이후 붙이면 됨.

**트레이드오프**
- 개발자 타겟인데 GitHub 로그인이 없는 아이러니. Alpha에서 유저 요청 있으면 즉시 추가 (오히려 서사가 강해짐).

**재검토 조건**
- Alpha에서 Google 로그인 마찰이 확인되거나 GitHub 요청이 나오면 즉시 GitHub 추가.

**면접 답변 요지**
> "MVP 진입 마찰을 낮추려고 Google 하나로 시작했다. Alpha 유저 피드백을 근거로 GitHub 확장 여부를 결정하는 게 스택 나열보다 좋은 서사라 판단했다."

### 2-7. OpenAI 단일 (gpt-4o-mini + text-embedding-3-small)

**대안과 이걸 택한 핵심 이유**
- Anthropic 단일: Anthropic엔 임베딩 API가 없어 임베딩을 딴 provider에서 받아야 함 → "단일 provider 회계" 이점이 무너짐. (핵심: Anthropic 임베딩 부재)
- 혼용(요약 Anthropic + 임베딩 OpenAI): rate limit·비용 회계가 2배로 복잡해져 F섹션 학습 목표(예산 정책 단순화)와 배치. (핵심: 회계 단순성)
- 자체 호스팅(BGE-M3 등): alpha 규모에 GPU·운영 복잡도가 과함. 임베딩 비용 월 예산 30% 초과 시로 미룸. (핵심: 규모 대비 운영 과함)

**근거**
- Spring AI OpenAI 어댑터가 가장 성숙해 통합 비용이 낮다.
- `text-embedding-3-small`은 1536차원·비용 균형 최적. pgvector에서 자연스럽게 다룰 수 있는 차원 수.
- `gpt-4o-mini`는 링크 3줄 요약에 품질 충분, 비용 낮음.
- 단일 provider면 rate limit·비용 회계(CS 매트릭스 F섹션 213~224L)가 단순.

**트레이드오프**
- Anthropic vs OpenAI provider fallback 실습 증거는 약함 → F섹션의 dependency fallback은 "OpenAI 장애 시 keyword search 유지"로 대체.
- 임베딩 provider 종속 → 임베딩 모델 교체 시 전체 재색인 필요 (CS 매트릭스 E섹션 128L에 이미 반영).

**재검토 조건**
- 월 예산 대비 임베딩 비용 30% 초과 시 자체 호스팅(BGE-M3, KURE 등) 실험.

**면접 답변 요지**
> "provider 하나로 rate limit·비용 회계를 단순화하고 CS 학습 목표(예산 정책 70/90/100%)에 집중했다. provider fallback은 keyword search 유지로 대체 실습한다. 임베딩 비용이 월 예산의 30%를 넘으면 자체 호스팅 실험을 트리거하는 조건도 정해두었다."

### 2-8. Spring REST Docs (+ restdocs-api-spec)

> **변경 이력:** 처음엔 SpringDoc OpenAPI를 "신입 개발 속도"로 택했으나, 이 프로젝트의 핵심 축이 "문서=구현을 CI로 보장"이라 **테스트로 검증되는 REST Docs**로 뒤집었다. 개발 속도는 가장 약한 근거였고, 프로젝트의 계약 테스트 철학과 정면으로 어긋났다.

**대안과 이걸 택한 핵심 이유**
- SpringDoc OpenAPI(annotation 기반): 어노테이션과 실제 구현이 어긋나도 문서는 그대로 생성됨 → 문서 신뢰도가 낮음. REST Docs는 **통합/계약 테스트가 통과해야 스니펫이 나와** "문서=구현"이 보장되고, 이는 B섹션의 "구현·문서 불일치를 CI에서 실패시킨다" 원칙과 정확히 맞음. (핵심: 테스트로 검증된 문서 신뢰도)
- 병용(REST Docs + SpringDoc Swagger UI): 즉시 눌러보는 상호작용 UI가 필요하면 SpringDoc으로 Swagger UI만 곁들일 수 있음. 다만 **계약의 SSOT는 restdocs-api-spec이 뽑는 OpenAPI 하나로 고정**하고 UI는 보조로 둔다. (핵심: 계약 SSOT 단일화)

**근거**
- REST Docs는 테스트가 통과할 때만 문서 스니펫을 생성한다 → 문서가 낡을 수 없다 (테스트 통과 = 문서 통과).
- `restdocs-api-spec`을 얹어 OpenAPI 3 산출물을 출력하면 B섹션의 "OpenAPI를 API의 SSOT로" 원칙을 그대로 유지한다.
- MockMvc·Testcontainers·TDD 흐름과 자연스럽게 붙어, "AI가 만든 코드를 테스트·계약으로 통제한다"는 이 프로젝트 서사를 강화한다.
- 국내 백엔드 실무(우아한형제들 등)에서 문서 신뢰도를 이유로 REST Docs를 선호하는 사례가 많아 채용 서사와도 정합.

**트레이드오프**
- 초기 작성 비용이 SpringDoc 어노테이션보다 큼(엔드포인트마다 문서화 테스트 필요) → 핵심 API부터 점진 적용.
- 즉시 탐색용 Swagger UI 편의는 SpringDoc이 나음 → 필요 시 UI만 병용.

**재검토 조건**
- 문서화 테스트가 개발을 과도하게 지연시키면 비핵심 API는 SpringDoc 어노테이션으로, 핵심 계약만 REST Docs로 이원화.

## 3. MVP 스코프: 단일 Link + 사용자 카테고리

MVP의 저장 단위는 하나의 **Link**다. 콘텐츠형·레퍼런스형·도구형을 별도 도메인 타입으로 분리하거나 사용자에게 선택받지 않는다. 사용자는 `레퍼런스`, `도구`, `Backend`처럼 카테고리를 직접 관리하고 하나의 Link에 여러 카테고리를 연결한다.

**결정 근거**
- 세 유형의 분기 규칙보다 저장→수집 시도→요약→임베딩→검색→연관 추천이라는 하나의 vertical slice를 먼저 검증하는 편이 제품 범위를 지킨다.
- 도구 사이트나 수집이 어려운 페이지도 URL·제목·OG 설명만으로 저장·카테고리·기본 검색이 가능하다. 본문·요약·임베딩은 가능한 경우에만 추가한다.
- 유형 자동 분류 LLM 호출과 유형별 예외 처리를 제거해 비용·rate limit·실패 표면을 줄인다.

**불변 규칙**
- URL과 fallback 제목은 저장 성공의 최소 보장이다.
- 사용자가 수정한 제목·요약은 재수집·재요약이 덮어쓰지 않는다.
- 카테고리 삭제는 Link를 삭제하지 않고 연결만 해제한다.

**재검토 조건**
- 실제 사용에서 특정 성격의 Link가 반복적으로 다른 파서·요약·다이제스트 규칙을 요구하면 내부 `linkKind`와 특화 파이프라인을 ADR로 검토한다.

## 부록. 기각한 후보와 이유 (간단)

| 후보 | 기각 이유 |
|---|---|
| Qdrant Cloud (별도 저장소) | 삭제 전파·인프라 표면 증가. pgvector로 통합 |
| RabbitMQ | 다중 컨슈머 독립 소비 불가. 지금은 아예 Job polling으로 시작하므로 무관 |
| Kafka (0주차부터 도입) | MVP 요구조건 아님. Job polling → Outbox → CDC의 3단계 진화로 도입 판정 |
| Netty·SSE | polling으로 충분. 실시간 요구 없음 |
| Cursor (하네스로) | Claude Code·Codex 데스크탑 앱이 자율 에이전트 성격이 같아 비교 실험 정합성 높음. Cursor는 IDE 내장형이라 결이 다름 |
| Anthropic 임베딩 | Anthropic은 임베딩 API 없음. 혼용은 rate limit·비용 회계 복잡도 2배 |
| MyBatis | 국내 채용 신호 감소 추세. QueryDSL로 유사 효과 |
| Spring MVC + Thymeleaf | Next.js가 채용 시장 신호 강함. 프론트 실습 증거도 필요 |
| Self-host LGTM (0주차부터) | 6GB 인스턴스 자원 압박. Grafana Cloud 무료로 시작 후 이관 옵션 |
| AWS SES (초기부터) | 샌드박스 이탈·도메인 검증 관리 복잡. Resend로 시작 후 규모 나면 이관 |
