# ADR-001: Modular Monolith (MSA 아님)

- 날짜: 2026-07-12 / 상태: 확정
- 출처: `learning/cs-learning.md`에서 분리 (0절)

## 상황

MVP 단계에서 서비스 경계와 독립 확장 요구가 아직 검증되지 않았다. `archive`, `ingestion`, `retrieval`, `generation`이라는 도메인은 있지만, 이들을 지금 물리적으로 분리할 근거(부하 격리·독립 배포·팀 경계)가 실측되지 않았다.

## 선택지

| 옵션 | 장점 | 단점 |
|---|---|---|
| **Modular Monolith** | 하나의 배포 단위, 낮은 운영 복잡도, 경계는 패키지·포트로 검증 | 물리적 격리 없음, 잘못 두면 경계가 무너짐 |
| MSA | 독립 배포·확장·장애 격리 | 네트워크·분산 트랜잭션·관측 복잡도, 미검증 경계를 조기 고착 |

## 결정

**Modular Monolith.** 하나의 배포 단위 안에서 `archive`, `ingestion`, `retrieval`, `generation`의 의존 방향과 데이터 소유권을 패키지로 분리한다.

## 트레이드오프

- 물리적 장애 격리가 없어 한 모듈의 문제가 프로세스 전체에 영향 가능 → bulkhead·queue로 프로세스 내 격리.
- 경계가 코드 규율에만 의존 → 패키지 의존 방향 테스트로 강제.

## 재검토 조건 (서비스 분리를 검토할 신호)

- 특정 모듈만 독립적으로 확장해야 비용 또는 SLO를 만족한다.
- 장애 격리가 한 프로세스 안의 bulkhead·queue로 해결되지 않는다.
- 배포 주기·소유 팀·보안 경계가 독립되어야 한다.
- 분리로 얻는 효과가 네트워크·분산 트랜잭션·관측 복잡도보다 크다는 실측이 있다.

## 면접 답변 요지

> "MSA를 몰라서 안 썼다"가 아니라 "경계를 패키지와 포트로 먼저 검증하고, 분리 신호를 사전에 정의했다."
